Circle Support Tool Summary

The Circle Support Tool behaves like a small parametric CAD feature. The support group remembers how it was created, while the individual support entities are treated as generated output. This is the key pattern to reuse for future tools.

When the user selects the Circle Support Tool in Support Mode, the Tool Options panel opens with Circle Support settings. The user clicks a first point on the selected model surface. That first point must be a real mesh hit. After that, mouse movement defines the second diameter point and updates a live preview circle plus generated support locations.

The second point can now be placed either on the model or off the model. If the mouse is off the model, the tool projects the cursor onto a horizontal construction plane through the first point. The circle itself remains horizontal, parallel to the XY plane. Supports are still only generated where the circle’s guide points can vertically project back onto the selected mesh.

Once the user clicks the second point, the circle preview locks. Moving the mouse no longer changes the circle. When the user clicks Apply, the generated supports are committed into a new SupportLayerGroup, the Tool Options panel closes, Circle Support mode exits, and all preview geometry disappears.

The main implementation pieces are:

CircleSupportOperation.cs handles mouse input, preview generation, Apply, Cancel, editing, and off-model second point projection.
ManualSupportTool.cs owns the currently active support operation and switches tools cleanly.
ToolOptionsPanel.xaml.cs exposes Circle Support settings such as spacing.
MainWindow.WorkspaceModes.cs connects UI events, selected layers, support mode, and tool activation.
SupportLayerGroup.cs stores the support group plus optional generator metadata.
CircleSupportSettings.cs stores the feature definition: first diameter point, second diameter point, and spacing.
SupportGroupGeneratorKind.cs identifies which tool generated a support group.
UpdateCircleSupportGroupCommand.cs makes editing an existing generated group undoable.
GphDocumentSerializer.cs persists generator metadata into .gph files.
SceneManager.cs and SupportRenderer.cs handle temporary transparency while editing.
Layer Behaviour

When a Circle Support group is created, it is added as a support layer group. The child supports remain individual SupportEntity objects, so they can render, hide, export, and participate in layer visibility like normal supports.

The group also stores metadata saying:

GeneratorKind = SupportGroupGeneratorKind.CircleSupport
CircleSupportSettings = first point, second point, spacing
This is important because the app does not try to infer the original circle from the generated supports. Instead, the support group remembers the original feature definition.

That is how real CAD systems usually work: a feature owns parameters, and geometry is regenerated from those parameters.

Editing Behaviour

When the user selects a Circle-generated support group in the Layer Panel, the app recognises the generator metadata and reopens the Circle Support settings in the Tool Options panel.

The saved spacing is loaded into the panel. The saved diameter points are reused as the circle definition. When the user changes spacing, the preview updates in real time using the stored circle and edited spacing.

While editing an existing Circle Support group, the old committed supports are made 50% transparent. This lets the user see the previous result while comparing the live regenerated preview.

When Apply is clicked during editing, the app updates the same support group rather than creating a new one. The old generated supports are replaced with the newly generated supports, and the updated settings are saved back onto the same group.

Undo/redo is handled by an explicit command. The command stores the old settings, old support entities, new settings, and new support entities. Undo restores the previous metadata and supports. Redo reapplies the new metadata and supports.

Manual edits inside a Circle-generated group are intentionally overwritten when the group is regenerated.

Pattern For Future Tools

A future Line Support Tool should follow the same pattern:

Add a settings class, for example LineSupportSettings, in Pillar.Core.
Add a generator kind, for example SupportGroupGeneratorKind.LineSupport.
Store the line tool’s feature definition on SupportLayerGroup.
Create a rendering operation, for example LineSupportOperation, to handle clicks, preview, Apply, Cancel, and editing.
Create an undoable update command, for example UpdateLineSupportGroupCommand.
Persist the settings in GphDocumentSerializer.
Update the Layer Panel selection flow so selecting a generated line group reopens the correct Tool Options UI.
Make Apply update the existing group when editing, or create a new group when creating.
Clear previews and exit the tool after successful Apply.
The main rule is: generated supports are output, not the source of truth. The source of truth is the tool settings saved on the support group.

Common Pitfalls

Avoid inferring tool settings from support positions. That becomes fragile once spacing, skipped projections, mesh holes, or future settings are involved.

Avoid putting tool logic directly into MainWindow. MainWindow should coordinate UI flow, but the operation class should own interaction behaviour.

Avoid coupling core layer metadata to rendering types. The core project should know about settings and support groups, not Helix or viewport previews.

Avoid regenerating committed supports during every mouse move. Use transient preview geometry while editing, then commit only on Apply.

Avoid creating a new support group when editing an existing generated group. Editing should preserve the same layer identity.

CAD Architecture Principle

The Circle Support Tool now works like a simple parametric feature:

User input -> saved feature settings -> generated support entities -> undoable regeneration

That same model should guide future tools like line, grid, curve, array, or pattern supports.
